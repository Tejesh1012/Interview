---

## 🔹 Step 1: Create a sample Delta table

Let’s simulate a large sales dataset:

```sql
CREATE TABLE sales_delta (
    order_id BIGINT,
    customer_id BIGINT,
    product_id BIGINT,
    order_date DATE,
    amount DECIMAL(10,2)
) USING DELTA
PARTITIONED BY (order_date);

-- Load some dummy data
INSERT INTO sales_delta
SELECT
  id AS order_id,
  RAND() * 1000000 AS customer_id,
  RAND() * 1000 AS product_id,
  DATE_ADD('2024-01-01', CAST(RAND()*365 AS INT)) AS order_date,
  RAND() * 500 AS amount
FROM RANGE(10000000); -- 10 million rows
```

---

## 🔹 Step 2: Query without optimization

```sql
-- Query: Find all orders for customer_id = 12345
EXPLAIN COST


SELECT * FROM sales_delta WHERE customer_id = 12345;
```

👉 Likely result:

* **Scan files**: Many (because customer\_id is not a partition key).
* **Data scanned**: Could be GBs.

---

## 🔹 Step 3: Compact files with OPTIMIZE

```sql
OPTIMIZE sales_delta;
```

Now, rerun the same query:

```sql
EXPLAIN COST
SELECT * FROM sales_delta WHERE customer_id = 12345;
```

👉 Result:

* Fewer files scanned (metadata overhead reduced).
* But still scanning most of the table because rows are scattered.

---

## 🔹 Step 4: Apply ZORDER

```sql
OPTIMIZE sales_delta
ZORDER BY (customer_id);
```

Now run the query again:

```sql
EXPLAIN COST
SELECT * FROM sales_delta WHERE customer_id = 12345;
```

👉 Result after ZORDER:

* Only a few files scanned (because rows with `customer_id = 12345` are colocated).
* **Data scanned drops from GBs → MBs**.
* Query performance improves massively. 🚀

---

### 📊 Rough Performance Comparison

* **Before OPTIMIZE**: 2000 files, 20 GB scanned.
* **After OPTIMIZE**: 20 files, 20 GB scanned. (Fewer files but same data)
* **After OPTIMIZE ZORDER**: 20 files, \~50 MB scanned. (Smart clustering)

---

✅ **Key takeaway**:

* Use **`OPTIMIZE`** regularly to compact files.
* Use **`ZORDER`** when queries frequently filter on **non-partitioned columns** (like `customer_id`, `product_id`, etc).

### 1. Import Spark & Functions

```python
from pyspark.sql import SparkSession
from pyspark.sql.functions import col
import random
```

* **SparkSession** → entry point to Spark (you already have `spark` in Databricks by default).
* **col** → used to refer to DataFrame columns in expressions.
* **random** → Python’s random number generator (to create fake data).

---

### 2. Create Synthetic Data

```python
data = [(i, f"cust_{random.randint(1,100)}", random.randint(1,1000)) for i in range(1, 100001)]
```

👉 This is a **list comprehension** in Python that generates 100,000 rows.
Each row has 3 fields:

* `i` → just a unique ID (1 to 100000).
* `f"cust_{random.randint(1,100)}"` → generates customer IDs like `cust_1`, `cust_2`, … up to `cust_100`.
  (each row randomly gets one of 100 possible customers).
* `random.randint(1,1000)` → a random amount (1–1000).

✅ Example of first few rows:

```
(1, "cust_42", 500)
(2, "cust_17", 230)
(3, "cust_42", 999)
...
```

---

### 3. Create a DataFrame

```python
df = spark.createDataFrame(data, ["id", "customer_id", "amount"])
```

* Converts the Python list `data` into a **Spark DataFrame** with 3 columns:

  * `id` (int)
  * `customer_id` (string)
  * `amount` (int)

This is now distributed and managed by Spark.

---

### 4. Write Data to Delta with Many Small Files

```python
df.write.format("delta").mode("overwrite").option("maxRecordsPerFile", 1000).save("/mnt/delta/sales_delta")
```

Here’s what each part does:

* `.format("delta")` → store it as a **Delta Lake table**.
* `.mode("overwrite")` → if the folder already exists, replace it.
* `.option("maxRecordsPerFile", 1000)` → force Spark to split into **small files**, each containing \~1000 rows.

  * Since we have 100,000 rows → this creates \~100 files.
* `.save("/mnt/delta/sales_delta")` → write to the storage path `/mnt/delta/sales_delta`.

👉 This simulates the **small files problem** you’d normally see in real pipelines (e.g., streaming writes).
