# ✅ **1. How RocksDB Works Internally (Autoloader Engine)**

Autoloader uses **RocksDB** as a **local embedded key-value store** to track file ingestion state.

### ✅ What Autoloader stores in RocksDB:

Autoloader stores metadata for each file:

| Key                 | Value                          |
| ------------------- | ------------------------------ |
| file_path           | checksum, size, modified time  |
| file_id             | processed = true               |
| folder_scan_pointer | last scanned offset            |
| schema_snapshot     | minimal schema info (optional) |

### ✅ Purpose:

* Detect **new files only**
* Avoid re-reading old files
* Maintain state even if cluster stops

### ✅ How RocksDB prevents reprocessing:

When a new batch starts:

1. Autoloader scans storage directory
2. For each file, it checks RocksDB

   * If file exists → skip
   * If not → process it
3. After processing, the file metadata is committed back to RocksDB

📌 **Even if job restarts, files are never reprocessed.**

# ✅ **2. What DLT Checkpoints Store**

DLT uses **its own checkpoints**, NOT RocksDB.

Location looks like:

```
/pipelines/<pipeline-id>/checkpoints/
```

Inside each table’s folder:

✅ `_commits` → microbatch completion logs
✅ `sources/` → inbound offsets
✅ `state/` → streaming state store
✅ `metadata/` → table + pipeline metadata
✅ `watermark/` → event-time watermark info

### ✅ Purpose:

* Track progress for Bronze → Silver → Gold
* Guarantee exactly-once data propagation
* Store intermediate state for aggregations
* Manage incremental logic for every layer

### ✅ Example:

Silver checkpoint tracks:

* Last Bronze batch it processed
* Watermark (if defined)
* Streaming offsets
* Deduplication state (if using dropDuplicates)

Gold checkpoint tracks:

* Aggregation results (state store)
* Last Silver batch read

# ✅ **3. What Delta Lake Stores in `_delta_log`**

Every Delta table (Bronze, Silver, Gold) contains:

```
/mnt/delta/table/_delta_log/
```

Inside you find:

* `00000000000000000000.json` (first commit)
* `00000000000000000001.json` (next commit)
* `_commits` folder (optional)
* `_last_checkpoint` file

### ✅ What’s inside these JSON files?

Example commit log:

```json
{
  "add": {
    "path": "file1.parquet",
    "size": 2049,
    "modificationTime": 1690111222333,
    "dataChange": true
  }
}
```

Schema part:

```json
{
  "metaData": {
    "id": "c741e907",
    "format": "parquet",
    "schemaString": "{\"type\":\"struct\",\"fields\":[ ... ]}",
    "partitionColumns": [],
    "configuration": {}
  }
}
```

### ✅ This is where:

✅ Schema evolution
✅ Partition changes
✅ Table metadata updates
✅ Constraints
✅ Optimization operations
✅ VACUUM logs

are all stored.
