1.You are working with billions of records daily. Why would you use Databricks over ADF for processing such data?
ADF is mainly for orchestration — scheduling, monitoring, and moving data. But when we need large-scale transformations, streaming, or machine learning workloads, ADF alone isn’t efficient. Databricks, built on Spark, provides distributed compute for billions of records, real-time streaming, and advanced ML models. So, I would use ADF to orchestrate pipelines, but Databricks to do the heavy-lift transformations.



2.Can you explain the difference between All-purpose clusters and Job clusters in Databricks?
An All-purpose cluster is long-running and typically used for interactive development, notebooks, and collaboration among multiple users. It’s convenient but more expensive if kept running.
A Job cluster is created on-demand for a specific job, runs until the job finishes, and then auto-terminates, which makes it cost-effective and more suitable for production pipelines.



3.What is the Medallion Architecture in Databricks, and why is it useful?
The Medallion Architecture organizes data into progressive quality layers.
Bronze: Raw ingestion (landing zone, minimal transformations).
Silver: Cleaned, conformed data with joins, deduplication, and historical tracking.
Gold: Curated, business-level aggregates tailored for analytics, dashboards, or ML.
This layered approach improves data quality, scalability, and reusability, and makes pipelines easier to manage.*

If asked “why useful?” → emphasize quality improvements per layer and decoupling raw from business logic.



4.What’s the difference between Delta Lake and Parquet?
Parquet is a columnar storage format that’s efficient for reading and querying but is essentially static — it doesn’t support updates or deletes well.
Delta Lake is built on top of Parquet, but it adds ACID transactions, schema evolution, time travel, and streaming support. This makes it more suitable for incremental data pipelines, data lakehouse architectures, and reliable analytics.




5.HOW WOULD YOU IMPLEMENT SLOWLY CHANGING DIMENSION (SCD) TYPE 2 IN DATABRICKS DELTA TABLES?
IN SCD TYPE 2, WE KEEP HISTORY BY INSERTING A NEW ROW WHENEVER A DIMENSION RECORD CHANGES, WHILE MARKING THE OLD RECORD AS EXPIRED. TYPICALLY, WE USE COLUMNS LIKE EFFECTIVE_DATE, END_DATE, OR VERSION NUMBER.

IN DATABRICKS, THIS CAN BE IMPLEMENTED WITH A MERGE INTO COMMAND ON A DELTA TABLE:

WHEN MATCHED AND DATA HAS CHANGED → UPDATE THE EXISTING ROW’S END_DATE.

WHEN NOT MATCHED → INSERT A NEW ROW WITH THE LATEST VALUES AND A NEW VERSION/TIMESTAMP.*





6.What are some optimization techniques you’ve used in Databricks to improve performance?
Some key optimization techniques I use in Databricks include:
Predicate Pushdown → apply filters early so less data is scanned.
Projection Pruning → select only required columns.
Broadcast Joins → broadcast small tables to avoid large shuffles.
Partitioning & Bucketing → organize data for efficient reads/writes.
Z-Order Clustering → optimize Delta Lake files for faster queries.
Caching → cache reused DataFrames.
OPTIMIZE & VACUUM → compact small files and clean up old snapshots.
Tuning shuffle partitions (spark.sql.shuffle.partitions).
Always mention real-world impact → e.g., “I used Z-Ordering on customer_id and reduced query time from minutes to seconds.”





7.What is Auto Loader in Databricks, and why would you use it over a normal file load?
Auto Loader is a feature in Databricks for incremental ingestion from cloud storage (like ADLS, S3, GCS). It automatically detects new files only, scales to handle billions of files, and supports schema inference and evolution.
Compared to a normal file load, Auto Loader avoids reprocessing old files, reduces cost, and makes ingestion pipelines more efficient and reliable.
Always say why better → “avoids reprocessing old files” = big plus





8.Can you explain the difference between a Data Lake, Delta Lake, and Data Warehouse?
A Data Lake stores raw data in its native format, usually schema-on-read, which makes it flexible but not reliable for complex analytics.
A Delta Lake sits on top of a data lake and adds ACID transactions, schema enforcement & evolution, and time travel, making it reliable for both batch and streaming workloads.
A Data Warehouse is a curated, structured environment with schema-on-write, optimized for BI and reporting, but less flexible for semi-structured or unstructured data.

Tip: If they ask “which would you choose?” → say:
Data Lake for raw/cheap storage.
Delta Lake for processing + reliability.
Data Warehouse for BI/analytics consumption.





9.What is Unity Catalog in Databricks, and why is it important?
Unity Catalog is a centralized data governance solution in Databricks. It provides fine-grained access control, data lineage tracking, auditing, and secure data sharing across multiple Databricks workspaces and even across clouds.
This ensures consistency, compliance, and easier collaboration when managing data assets at scale.

Tip: If asked “why important?” → highlight compliance and security (many companies want to hear this).





10.Can you explain Spark architecture (driver, executors, cluster manager) in simple terms?
In Spark architecture:

Driver: The “brain” of the application. It runs on the master node, converts code into tasks, and coordinates execution.

Executors: Worker processes that actually execute the tasks and return results to the driver. Each executor has its own memory and CPU.

Cluster Manager: Allocates resources for the driver and executors (e.g., Databricks, YARN, Kubernetes).

Together, the driver plans the work, executors perform it, and the cluster manager ensures resources are allocated.
Tip: Keep it simple → “Driver = plans, Executors = work, Cluster Manager = resource allocator.”





Q11. What is Predicate Pushdown and Projection Pruning in Spark, and why are they useful?
Predicate Pushdown means Spark pushes filters down to the data source so only the relevant rows are read, reducing I/O.
Projection Pruning means Spark only reads the required columns, instead of the entire dataset.

Both improve performance by scanning less data, reducing memory usage, and speeding up queries*.

For example, instead of reading 100 GB, Spark only scanned 5 GB after pushdown, reducing job time from 10 minutes to 2 minutes.





Q12.How would you design a fault-tolerant pipeline in Databricks/ADF to handle billions of records daily?
To design a fault-tolerant pipeline for billions of records daily in ADF + Databricks, I would:

Ingestion: Use Auto Loader or partitioned copy activities with retry policies, checkpointing, and idempotent writes to handle retries without duplication.

Scalability & Parallelism: Partition data by natural keys (e.g., date, region) and process in parallel across executors.

Storage: Land raw data first in a Bronze layer (immutable storage) so that failed runs can be reprocessed.

Processing: Use Delta Lake for ACID guarantees, schema evolution, and exactly-once writes.

Resilience: Enable checkpointing in streaming, so jobs resume from the last processed offset.

Monitoring & Alerting: Set up ADF alerts (failure notifications) + Databricks job monitoring dashboards.

Recovery: Implement dead-letter paths for bad records, and a replay mechanism to reprocess only failed batches instead of the entire dataset.

This ensures reliability, scalability, and fast recovery without data loss or duplication.

Tip: The magic words interviewers look for here are:

Checkpointing
Idempotency
Delta Lake ACID guarantees
Dead-letter handling
Reprocessing strategy (Bronze layer)
