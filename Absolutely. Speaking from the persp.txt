## 🚧 Common Issues Across S3 → Glue → Redshift Pipelines

---

### 📁 **1. Data Issues in S3**

| Problem                     | Description                                                                 |
| --------------------------- | --------------------------------------------------------------------------- |
| ❌ Inconsistent schema       | Same file name, different structure — breaks Glue crawlers                  |
| ❌ Bad partitioning          | Missing or wrongly formatted partition folders like `dt=2024-13-99/`        |
| ❌ Header rows in every file | Glue misinterprets headers as data                                          |
| ❌ Compression mismatch      | Files are `.gz` but read as `.csv`                                          |
| ❌ Encoding errors           | Files saved in `UTF-16` or malformed characters (often with 3rd party logs) |

> 🔧 **Fix**: Enforce schema validation at ingestion (e.g., AWS Lambda or PySpark in EMR/Glue job), and set up file format + naming standards.

---

### 🧠 **2. Glue Crawler & Catalog Issues**

| Problem                             | Description                                                                      |
| ----------------------------------- | -------------------------------------------------------------------------------- |
| 🔁 Crawler keeps overwriting schema | One wrong file causes the schema to change or break queries                      |
| ❌ Tables not appearing in Redshift  | Glue table created, but not in same region or IAM role not permitted             |
| 🐌 Slow crawling                    | Due to too many small files in deeply nested S3 folders                          |
| 🔒 Permission Denied                | Role used by Redshift doesn't have `glue:GetTable` or `s3:GetObject` permissions |
| 🛑 Crawler fails silently           | No notification, logs hidden in CloudWatch unless explicitly set up              |

> 🔧 **Fix**: Use partitioned S3 layout, schedule crawlers selectively, and always lock schema after crawler run in prod.

---

### 🛠️ **3. Glue Job Failures**

| Problem                       | Description                                                                       |
| ----------------------------- | --------------------------------------------------------------------------------- |
| ⏱️ Timeout on large datasets  | Default job timeout too low, or data skew causes certain partitions to overload   |
| ❌ PySpark memory errors       | e.g., `OutOfMemoryError`, especially with wide records or joins                   |
| 🔀 Data type mismatch         | Glue jobs converting float to string silently, causing downstream Redshift issues |
| 🔒 Access Denied to S3        | IAM Role lacks `kms:Decrypt` if S3 data is encrypted                              |
| 💥 Partition overwrite issues | Using `overwrite` mode wipes out the whole partition unintentionally              |

> 🔧 **Fix**: Parameterize Glue jobs, profile data before processing, and always define partition overwrite behavior carefully.

---

### 🧬 **4. Redshift External Schema Issues**

| Problem                 | Description                                                                                      |
| ----------------------- | ------------------------------------------------------------------------------------------------ |
| ❌ Table doesn’t show up | External schema created, but Glue database name or region is wrong                               |
| 🔒 Permission errors    | IAM role not attached to Redshift or doesn't have `glue:GetDatabase`                             |
| 🐌 Poor performance     | Spectrum reads huge S3 files without partition pruning                                           |
| 📉 No stats             | External tables don’t benefit from Redshift optimizer unless `ANALYZE` is run on internal tables |

> 🔧 **Fix**: Pre-partition S3 properly, maintain clean Glue catalogs, and monitor usage with Redshift query logs.

---

### 🔁 **5. End-to-End Pipeline Monitoring Challenges**

| Problem                        | Description                                                                                     |
| ------------------------------ | ----------------------------------------------------------------------------------------------- |
| ❌ Lack of alerts               | Jobs fail, but no CloudWatch alerts, so teams notice it days later                              |
| 🕳️ Silent data quality issues | e.g., one file has null `user_id` but flows all the way to reporting layer                      |
| 🤖 Missing automation          | Manual steps in deployment, or data refresh breaks when new partition isn’t added automatically |

> 🔧 **Fix**: Use **CloudWatch + SNS** for alerting, **Deequ or Great Expectations** for data quality, and **Step Functions or Airflow** for orchestration.

---

## ✅ Final Real-World Advice

| Area             | Pro Tip                                                                                               |
| ---------------- | ----------------------------------------------------------------------------------------------------- |
| 🧪 Testing       | Always test on sample partitions first before loading into Redshift                                   |
| 🔐 IAM           | Design least-privilege roles for Glue and Redshift separately                                         |
| 🏗️ Architecture | Keep raw → processed → curated data layers clearly separated in S3                                    |
| 📦 File Format   | Prefer **Parquet** or **ORC** over CSV for analytics performance                                      |
| 📊 Performance   | For heavy queries, **unload data** from S3 into native Redshift tables for fast joins and aggregation |

CREATE OR REPLACE PROCEDURE refresh_reporting_tables()
LANGUAGE plpgsql
AS $$
BEGIN
    -- Step 1: Drop old snapshot
    EXECUTE 'DROP TABLE IF EXISTS sch_reporting.sales_snapshot';

    -- Step 2: Recreate it with updated data
    EXECUTE 'CREATE TABLE sch_reporting.sales_snapshot AS
             SELECT region, SUM(sales) as total_sales, CURRENT_DATE as snapshot_date
             FROM sch_raw.sales_data
             GROUP BY region';

    -- Step 3: Run analyze for better performance
    EXECUTE 'ANALYZE sch_reporting.sales_snapshot';
END;
$$;

2. What does df = df.toDF() mean?
This line converts a DynamicFrame (used by AWS Glue) into a regular Spark DataFrame.

🔍 Why the Conversion?
Glue uses DynamicFrame as its own abstraction — good for semi-structured data and AWS-native operations.

But for powerful Spark operations like .groupBy(), .join(), .agg(), etc., you often need to use a regular PySpark DataFrame.
